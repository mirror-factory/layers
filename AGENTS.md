## AI SDK v6 Patterns (CRITICAL)

- Use `inputSchema` NOT `parameters` in tool definitions
- Use `toUIMessageStreamResponse()` NOT `toDataStreamResponse()`
- Message format: `message.parts[]` NOT `message.content`
- Tool part types: `part.type === 'tool-{toolName}'`
- Tool states: `input-streaming` | `input-available` | `output-available` | `output-error`
- `addToolOutput` NOT `addToolResult`
- `sendMessage` NOT `append`
- `convertToModelMessages()` must be `await`ed (async in v6)
- Multi-step server: `stopWhen: stepCountIs(n)` NOT `maxSteps`
- Generator tools: last `yield` is the final output (not `return`)

- **Product spec**: before large product/app work, read `.ai-starter/product-spec/latest.md`; run `pnpm product:spec` to interview, agent-fill, or explicitly bypass the YC-style spec
- **Product validation**: before large product/app work, read `.ai-starter/product-validation/latest.md`; if missing or incomplete, run `pnpm product:validate` or explicitly mark the validation bypass reason
- **MFDR**: for product/architecture/API/tool/UI direction, read `.ai-starter/mfdr/latest.md`; run `pnpm mfdr` when the hypothesis, API/service choice, UI approach, or verification strategy changes
- **Design**: read `DESIGN.md` and `.ai-starter/manifests/design.json` before UI changes; do not let generated UI drift from the declared visual contract
- **Planning**: use `pnpm plan -- "<task>"` or update `.ai-starter/plans/latest.json` before feature-sized writes
- **Context**: read `.ai-starter/manifests/*`, existing source patterns, and local docs before generating or editing
- **Runtime state**: Codex uses `.codex/config.toml` + `.codex/hooks.json`; Claude Code uses `.claude/settings.json`; both share `.ai-starter/*`
- **Research**: check `.ai-starter/research/` for cached library docs before dependency, framework, provider, or browser automation work
- **Verification**: run `pnpm sync`, `pnpm score`, typecheck/tests, and browser proof before handoff
- **Errors**: Check server logs for `CHAT ROUTE ERROR:` stack traces first

## Release Flow (READ BEFORE PUSHING)

**Never push directly to `main`.** Every change reaches `main` via a PR from `staging`, which itself received the change via a PR from `development` (or a feature branch). Single exception: a tagged hotfix from a `hotfix/*` branch with an explicit incident write-up linked in the PR.

Order of promotion:

```
feature/* → development → staging → main
```

| Branch        | Domain                            | Vercel env       | Stripe    |
| ------------- | --------------------------------- | ---------------- | --------- |
| `main`        | `layers.mirrorfactory.ai`         | Production       | Live      |
| `staging`     | `staging.layers.mirrorfactory.ai` | Preview (pinned) | Test      |
| `development` | `dev.layers.mirrorfactory.ai`     | Preview (pinned) | Test      |
| feature/\*    | per-PR Vercel preview             | Preview          | Test      |

Full setup, env-var checklist, GitHub branch protection, Vercel domain pinning, and the OAuth/webhook allow-lists live in [`docs/RELEASE.md`](./docs/RELEASE.md). When `staging` and `development` aren't wired up yet, follow [PROD-383](https://linear.app/mirror-factory/issue/PROD-383).
