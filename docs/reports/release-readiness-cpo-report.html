CPO Questions Answered

Question Answer Action
Are Apple and Android items for stores/TestFlight? Yes. Apple signing/notarization/TestFlight blockers are for distributable macOS/iOS artifacts. Android upload-key blockers are for Play internal testing or production release artifacts. They do not block local simulator/debug QA. Provide App Store Connect API key material, Developer ID/notarization credentials, and Android upload-key secrets before store-grade CI can be proven.
What do we do about Dependabot? Treat it as a promotion blocker, not a local simulator blocker. The default branch alerts should close only after the dependency updates land through development, staging, and main. After this branch is finalized, merge through the branch ladder, then rerun GitHub Dependabot alert checks on the default branch.
What about three-domain promotion? The target remains dev.layers.mirrorfactory.ai for development, staging.layers.mirrorfactory.ai for staging, and layers.mirrorfactory.ai for main. Vercel branch pinning exists, but DNS verification is still the missing piece. Add Cloudflare CNAME/TXT records, verify the Vercel domains, then prove development to staging to main with separate PRs.
Is Claude still being used? Yes, but not blindly. Claude was used for bounded web/platform checks. Where Claude stalled on iOS, Codex took over direct simulator testing and recorded the blocker. Continue using Claude for parallel review/evidence, but keep final release decisions tied to observed test output.
Is the report standardized? Yes. The report skill exists and this HTML file is the durable CPO update target. Future status requests should update docs/reports/release-readiness-cpo-report.html first, then summarize in chat.
